在关系型数据库的世界，要想让所有的查询都很迅速，是一个不小的工作量，其中索引的设计和优化占很大一部分。所以我们需要知道索引设计的基本原则(这些基本原则适应绝大多数关系型数据库)，对于某些特殊场景，可能还需要对存储引擎的某些参数进行调优。这篇文章主要记录下索引设计与优化的一些东西

在设计或优化查询语句时，我们需要时刻牢记以下两点：
* 站在优化器的角度看问题
* 很多情况下随机IO比临时排序更耗费时间

## 索引存在的意义
个人认为索引存在的意义只有一句话：尽最大可能减少磁盘的随机io

## 查询的类型
在我们工作中，主要接触到以下几个类型的查询：
* 简单谓词的查询
* 分页查询
* 多索引查询
* 连接查询
* 困难谓词
* UNION查询

文章后面慢慢会说到这这几类查询的情况

## 一些术语与简单背景
本文或索引优化过程中可能会涉及的一些术语：
* 半宽索引，where子句的所有字段都包括在一个索引中
* 宽索引，where子句的所有字段和查询所需的所有字段都包括在一个索引中，和覆盖索引差不多
* 聚集索引，或者聚簇索引，索引的叶子节点是直接指向数据页的
* TR，随机IO次数，假设一次随机IO访问的时耗为10ms
* TS，顺序IO次数，假设一次顺序IO访问的时耗为0.01ms
* FETCH，提取一行数据，假设提取一行数据的时耗为0.1ms
* 索引的三颗星，第一颗星：索引能命中一个比较窄的索引片(半宽索引)；第二颗星：索引能避免排序；第三颗星：索引能避免回表访问(宽索引或覆盖索引)
* 过滤因子，记为FF，和索引的选择性意思差不多，对应的数据记录在数据中的总行数
* 排序时间，现在排序的效率已经很高了，我们假设排序和顺序IO访问的时耗一致，为0.01

从概念上来说，半宽索引可以帮助命中一个很小的索引片；而宽索引可以避免回表访问，从而能间接避免随机IO访问

对于以上IO访问时间假设只是简单的估算，方便我们分析问题，不同的设备是不一样的；不过有一些可以肯定，在很长一段时间内，随机IO访问的时耗远远大于顺序IO的

聚簇索引的优点是： 通过顺序访问聚簇索引而引发的回表访问数据大部分情况都是顺序IO

## 评估查询的顺序IO和随机IO次数
虽然数据库引擎读取索引与数据时不一定全部都是从磁盘读取，我们假设它们每次是从磁盘读取的，因为这一过程对索引设计过程无伤大雅

在有索引的情况下，简化的数据访问的过程为：首先访问索引，第一次总是随机IO，其后看情况是随机IO还是顺序IO；再访问表，第一次总是随机IO，其后根据情况随机IO或顺序IO访问；最后将数据提取出来(包括从存储引擎层到server层)

举个例子:
```SQL
select * from EXAMPLE where A=100
```
假设A的FF为10%，整个EXAMPLE表里面有1000000行数据，且A不是聚簇索引，那么他们的IO次数是多少呢？

我们分两个层面来说： 索引和表访问
* 索引的TR: 1
* 索引的TS: 1000000*10%
* 表的TR: 1000000*10%
* 表的TS: 0
* 提取行数: 1000000*10%

如果将查询语句换成:
```SQL
select A from EXAMPLE where A=100
```
那么IO访问次数就变成：
* 索引的TR: 1
* 索引的TS: 1000000*10%
* 表的TR: 0
* 表的TS: 0

因为对于这个查询来说，索引A变成了覆盖索引，所以不需要回表访问

## 怎么设计最佳索引
针对某个查询语句，怎么设计最好的索引呢？首先我们肯定是尽量设计一个三星索引，如果因为某些原因，不能设计出一个三星索引，那么可能就需要在避免排序和选择近可能窄的索引片两者之间做选择了

对于一个特定的语句，基本的设计思路如下：
1. 一般情况下，我们都能设计出一个宽索引，即覆盖索引，当然这需要插叙语句的配合
2. 如果索引能够命中窄索引片和排序需求同时满足，那么对于一个特定的查询语句，这个索引的设计算是完成了
3. 如果第2步不能满足，那么我们需要选择将排序的字段放在索引的前面以避免排序；还是需要尽量命中一个窄的索引片来让数据库帮我们进行一次排序操作
4. 如果前面的步骤还不能得到一个比较好的响应时间，那么可能对表的设计进行调整，比如进行反范式的表设计

当然，索引的设计过程远不止这么简单，有以下几点决定了索引设计的复杂性：
* 一般应用中，查询不止一个，我们需要综合多个查询语句来考虑
* 有些复杂谓词，优化器可能有不同的访问路径选择
* 有些复杂的查询，优化器可能有不同的访问路径选择
* 数据量的不同或者数据的分布不同，优化器可能有完全不一样的选择

## 怎么评估一个索引的好坏
我们设计完一个索引了，或者在索引的设计过程中，我们怎么评估一个索引的好坏呢？有以下两个基本办法：
* 基本问题法，判断是否满足索引的第一颗星(半宽索引)，这是最基本的，如果还不能满足性能，那么需要看是否能够满足另外两颗星；这个方法的问题在于有些严重情况可能会漏掉
* 快速上限估算法，简称QUBE，用这个方法可以估算一个查询的响应时间，它的原则是以比较悲观的判断，会有一定误报，但是不会漏掉某些问题

我们主要说说快速上限估算法

### QUBE之半宽索引
语句:
```SQL
select * from EXAMPLE where A=100 and B=90
```
假设有个联合索引(A,B)，且不是聚簇索引，A的FF为10%，B的FF为1%，EXAMPLE表有1000000行数据：
* 索引，TR: 1，TS: 1000000*0.1*0.01
* 表，TR: 1000000*0.1*0.01，TS: 0
* FETCH，1000000*0.1*0.01

那么这个查询的响应时间为： (1+1000000*0.1*0.01)*10 + 1000000*0.1*0.01*0.01 + 1000000*0.1*0.01*0.1， 10120ms

### QUBE之宽索引
对于语句：
```SQL
select A, B from EXAMPLE where A=100 and B=90
```
* 索引，TR: 1，TS: 1000000*0.1*0.01 
* 表，TR: 0，TS: 0
* FETCH，1000000*0.1*0.01

那么这个查询的响应时间为： 1*10+1000000*0.1*0.01*0.01 + 1000000*0.1*0.01*0.1， 120ms

可见随机IO对查询时间的影响是很大的

### QUBE之限制结果集
那么对于以下语句又怎样呢？
```SQL
select A, B from EXAMPLE where A=100 and B=90 limit 20
```
* 索引，TR: 1，TS: 20
* 表，TR: 0，TS: 0
* FETCH，20

那么这个查询的响应时间为： 1*10 + 20*0.01 + 20*0.1， 12ms

### QUBE之范围查询的情况
对于语句：
```SQL
select A, B from EXAMPLE where A=100 and B>90 and B<100
```
假设：只有一个单列索引B，且不是聚簇索引，A的FF为1%，B的值为90-100的FF为10%，EXAMPLE表有1000000行数据

* 索引，TR: 1，TS: 1000000*0.1
* 表，TR: 1000000*0.1，TS: 0
* FETCH，1000000*0.01*0.1

那么这个查询的响应时间为： (1+1000000*0.1)*10 + 1000000*0.01*0.01 + 1000000*0.01*0.1，大约1000秒

### QUBE之过滤因子隐患
对于语句：
```SQL
select A, B, C from EXAMPLE where A=100 and B=90 order by C limit 20
```
假设：有个联合索引(B, C)，且不是聚簇索引，A的FF为10%，B的FF为1%，EXAMPLE表有1000000行数据

* 索引，TR: 1，TS: 20/10%
* 表，TR: 20/10%，TS: 0
* FETCH，20/10%

那么这个查询的响应时间为：201*10 + 200*0.01 + 200*0.01， 大约2s

如果前面的条件中A的FF为0%又如何呢？
* 索引，TR: 1，TS: 1000000*0.01
* 表，TR: 1000000*0.01，TS: 0
* FETCH，0

那么这个查询的响应时间为：10001*10 + 1000000*0.01*0.01， 大约100s

这就是过滤因子隐患的情况，在这种情况下，更窄的索引片宽度可能比避免排序更加重要

过滤因子隐患在以下三个条件同时满足的情况下可能就会产生：
* 访问路径中没有排序
* 限制返回行个数
* 不是所有字段都为匹配字段，即不是半宽索引

## 表连接情况下优化器的选择
在两表连接的情况下，优化器可能会选择以下三种方案之一：
* 循环嵌套连接，以其中一张表为外层表，另外一张表为内层表，相当于代码里的两个嵌套的for循环
* 合并扫描连接，两张表分开查找，然后合并两者的结果，相当于代码里面的三个独立的for循环
* 哈希连接，对较小的结果集用哈希算法计算其连接字段，并将其保存在临时表中；然后再扫描另外一个表，与临时表进行匹配

具体情况优化器会根据统计信息来决定选用哪种方式

对于循环嵌套连接，可能造成的问题是： 即使使用最优的访问路径和理想索引，对内层表的随机访问也会太多，解决这个问题的办法是添加冗余的列，使一张表上包含所有的本地谓词(反范式表设计)

## 多索引的情况
我们主要考虑or, and 连接多个索引的情况

在 and 连接两个索引时，数据库有可能会选择只利用其中一个，也可能两个都使用： 分别根据索引找到相应的数据行的指针，然后合并结果，这种情况我们最好能为它设计一个宽索引或半宽索引

在 or 连接两个索引时，数据库很大可能会选择全表扫描，也可能两个都使用：分别根据索引找到相应的数据行的指针，然后合并结果 

## 困难谓词
不同数据库的困难谓词是不一样的，候选列表有：in, or, <>, union, like, DISTINCT, having, not in, group by, exist 等，这些谓词有的是需要特殊处理的，有的是的确是困难谓词，它们的处理方式可能也和特定数据库相关

## 站在优化器的角度看问题
* 优化器路径选择的多样性，主要基于成本
* 连接方式的选择
* 多索引访问方式的选择
* 困难谓词的处理，各个数据库的困难谓词都不一样

## 总结
索引优化是个比较需要经验的课题，需要在实际工作中多思考，不过基本思路还是不变的：
* 尽量命中比较窄的索引片
* 尽量避免过多的随机IO
* 摸清楚优化器的优化套路
