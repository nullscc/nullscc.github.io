有些情况我们需要保证多个步骤要么都成功，要么都失败，这时候就要用到数据库的事务支持，虽然各个数据库的对于事务的支持情况不一，但是他们的基本原理都是类似的

## 事务的特性
ACID: 
* Atomicity, 原子性，事务是不可分割的步骤
* Consistency, 一致性，事务操作的数据是一致的，比如A给B转账，转账前后A与B的账户余额总数不变
* Isolation, 隔离性，两个并发的事务不互相干扰
* Durability, 永久性，事务一旦提交，提交后的数据不会丢失(即使机器突然断电了)

## 事务的隔离级别
事务的隔离级别描述的是： 如果两个事务并行，并且修改或读到同一段数据时的表现

### Read Committed(Nonrepeatable read)
* 没有脏读，只能读到已经提交的数据
* 没有脏写，两个事务并发写同一个数据，只能重写已经提交了的写数据，即需要等待正在写的另外一个事务提交

是 PostgreSQL 的默认隔离级别

#### 如何实现
两种实现方式
1. 每次写时都获取一个行级锁，直到事务整个事务完成以后才释放这个锁；每次读时也获取同一个锁，读完(不必等到事务完成)立即释放；这种方式的缺点是当有很多写时会阻碍读的事务
2. 每次写时不占用锁，让数据库记住事务写的新值和旧值，当另外一个事务读取时，仅当其他事务提交了才读取新值

#### 缺点
事务A和事务B，事务B中有两次读取一样的记录，事务A修改了事务B读取的记录，可能会导致事务B在事务A结束前后得到的数据不一样（例子： 银行转账，总数会临时不对）

这个缺点在某些情况不会造成什么影响，因为它是最终一致的，但是某些情况会有很大影响：
* 备份，如果恰好在读不一致的时候进行数据备份，那么这时候备份的数据就永远不对了
* 查询分析与一致性检查

### Snapshot isolation(Repeatable read)
可以解决 Read Committed 的缺点

每个事务开始的时候都保存一份快照，整个事务的读都只针对这个快照点的数据

PostgreSQL, MySQL支持

#### 如何实现
* 写时加锁，但是读时不加锁
* multi-version concurrency control (MVCC)，对每个数据记录都保持一个已提交事务的版

#### 缺点
* 在 read-modify-write 丢失与 compare-and-set 方面会导致一些方法失效(因为read和where时可能会读到快照的数据；where的情况视数据库实现而定)
* 幻影读

## 幻影读
在两个事务并行的过程中，两个事务均进行 检测 -> 执行写入 的过程，但是两个事务的写入会互相影响检测的结果(回读到实际已经失效的数据)

### 怎么解决幻影读？
由于幻影读从本质上来说是因为某些情况整个表的任意一个数据变动都能影响到某些查询的结果，所以需要做等同于锁表的操作，但是由于有些查询又是可以联表查询的，所以解决办法有限

* 物化冲突，需要在应用层做调整，基本原则是在将检测的查询换成一个另外的数据，相当于这个另外的数据充当了检测点，需要在应用层额外实现
* 完全解决的办法是：序列化隔离

## 预防写丢失
这是个应用层面的问题，read-modify-write场景：需要先读取出来数据，然后需要在读取出来的数据上进行更改，然后再把它写进去，有些情况会导致数据丢失，有几个办法可以解决

* 原子写，UPDATE counters SET value = value + 1 WHERE key = 'foo';
* 在事务中对读的操作进行加锁(FOR UPDATE)，这样别的事务会就必须等待这个事务完成，但是还是没办法保证 Snapshot isolation 隔离级别会发生这种情况
* 数据库系统检测这种情况，然后再次重试，PostgreSQL的 repeatable read 支持，但是MySQL貌似不支持

## 序列化隔离
原则是保证所有事务的执行是串行的，完全不存在并行的情况

### 如何实现
#### 顺序执行事务
* 存储过程
* 其他

#### 两段锁（Two-Phase Locking (2PL)）
原则： 
* 如果事务A已经读取了一个对象，那么如果事务B想要写那个对象，那么事务B需要等待事务A提交或者中断；
* 如果事务A已经写了一个对象，事务B想要读那个对象，那么必须等待事务A提交或者中断；
* 如果事务A与事务B都想要读同一个对象，那么没任何影响

实现：
* 共享锁： 可以多次获取
* 排他锁： 如果已经有排他锁被获取了，则必须等待释放；如果有共享锁获取了，那么也必须等待这个共享锁释放

再具体点：
* 如果事务想要读一个对象，那么必须获取共享锁，想要写一个对象，那么必须获取一个排他锁
* 如果一个事务先读取一个对象，后写一个对象，那么必须将锁从共享锁升级成排他锁
* 事务一旦获取锁，只能在事务提交或者中止时才能释放

两阶段锁的名称的由来： 第一阶段，发现写的情况，加排他锁；第二阶段，直到事务提交或者中止，释放排他锁

##### 缺点
* 性能问题
* 很容易造成死锁，需要有自动检测死锁的机制以便中止死锁的事务然后重试

#### Serializable Snapshot Isolation (SSI)
基于乐观并发控制原则： 假设不会有任何不好的事情发生，在提交的时候进行检测，如果检测没问题，则真正提交，如果检测有问题，则中止再次重试

它是基于 Snapshot isolation 隔离级别的，只不过在事务即将要提交是加了很多个检测算法

优点： 在某些场景下兼顾性能与序列化隔离

缺点： 在某些情况下可能会更加恶化，比如重试次数过多

## 思考
事务在带来安全的同时(虽然由于不同隔离级别的实现安全程度也不一样)，会严重影响性能，有时候业务设计时可以根据业务的特点，在应用层做一些措施，不一定非要强求事务

而且我们在应用事务的时候，不能想当然的认为它是没问题的，事务的安全程度取决于具体的实现(比如隔离级别)
